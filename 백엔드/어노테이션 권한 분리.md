# TIL: 커스텀 어노테이션으로 권한 분리하기

> Spring에서 권한 검증을 직접 만든 어노테이션으로 처리하는 이유

## 한 줄 요약

권한 검증 로직을 비즈니스 로직에서 떼어내기 위해(**관심사 분리**), 검증을 "표식"인 어노테이션과 그걸 가로채는 AOP/Interceptor로 분리한다.

## 왜 커스텀 어노테이션을 만드는가

### 1. 코드 중복 제거

매 메서드마다 권한 체크 코드를 넣으면 검증 로직이 곳곳에 흩어진다.

```kotlin
// Before: 검증 코드가 모든 메서드에 반복됨
fun deleteResume(resumeId: Long, userId: Long) {
    val resume = resumeRepository.findById(resumeId)
    if (resume.ownerId != userId) {
        throw ForbiddenException()
    }
    // 실제 로직
}
```

```kotlin
// After: 검증은 어노테이션으로 위임, 순수 로직만 남음
@CheckResumeOwner
fun deleteResume(resumeId: Long) {
    // 권한 검증은 이미 끝났다고 가정
}
```

### 2. 선언적(Declarative) 표현

"이 메서드는 작성자만 접근 가능"이라는 의도가 어노테이션 한 줄로 명확히 드러난다. 메서드 시그니처만 봐도 권한 정책을 파악할 수 있다.

### 3. AOP와의 결합

어노테이션은 **표식**일 뿐이고, 실제 검증은 이걸 가로채는 Aspect/Interceptor가 처리한다.

```kotlin
@Aspect
@Component
class ResumeCurrentUser {
    @Before("@annotation(CheckCurrentUser)")
    fun check(joinPoint: JoinPoint) {
        // 여기서 실제 권한 검증
    }
}
```

## Spring Security 기본 기능 대신 직접 만드는 경우

`@PreAuthorize("hasRole('ADMIN')")` 같은 게 이미 있는데도 커스텀 어노테이션을 만드는 이유:

- **도메인 특화 권한**: "ADMIN이냐"가 아니라 "이 게시물의 **소유자**냐", "이 강의의 **담당 교사**냐" 같은 리소스 단위 권한은 SpEL로 표현하기 복잡하고 지저분하다.
- **가독성**: 긴 SpEL 표현식보다 `@CurrentUserOnly` 같은 의미 있는 이름이 훨씬 읽기 좋다.
- **복잡한 검증 로직**: SpEL 한 줄에 못 담는 검증은 어차피 별도 컴포넌트가 필요하다.

## 구현 방식 비교

| 방식 | 특징 |
|------|------|
| AOP (`@Aspect`) | 메서드 레벨 가로채기, 가장 유연 |
| `HandlerInterceptor` | 컨트롤러 진입 전 처리, 웹 요청 단위 |
| `@PreAuthorize` 확장 | Spring Security 메서드 시큐리티 활용 |

## 정리

- 어노테이션 = **무엇을 검증할지** 선언하는 표식
- AOP/Interceptor = **어떻게 검증할지** 처리하는 실행부
- 둘을 분리하면 비즈니스 로직은 권한 코드에서 자유로워지고, 권한 정책은 한곳에서 관리된다.
