# TIL — 도메인과 DNS

> **한 줄 요약**: 사람이 외우기 쉬운 도메인 주소를, 컴퓨터가 통신에 쓰는 IP 주소로 바꿔주는 시스템이 DNS이며, 이 변환은 여러 계층의 DNS 서버를 단계적으로 거쳐 이루어진다.

---

## 도메인이란?

웹사이트에 접속하기 위해, 외우기 어려운 IP 주소 대신 사용하는 **문자열 형태의 인터넷 주소**다.

---

## DNS와 네임서버

**DNS**는 웹사이트의 IP와 도메인을 이어주는 시스템이고, 도메인의 IP를 찾아주는 서버가 **네임서버**이다.

DNS 서버는 "특정 도메인 주소 = 특정 IP 주소"라는 기록을 저장해 두고, 인터넷 사용자가 도메인 주소를 검색했을 때 해당 IP 주소로 연결해 준다.

**동작 예시**

1. 클라이언트가 브라우저에서 `NAVER.com`을 검색한다.
2. 요청이 DNS 서버로 전달되고, 서버 내부에서 `NAVER.com = 12.123.123.123` 항목을 찾아낸다.
3. 브라우저에게 "이 IP 주소를 가진 호스팅 서버(웹사이트 데이터가 저장된 곳)로 가라"고 안내한다.
4. 브라우저가 해당 IP로 접속해 웹사이트가 화면에 보인다.

---

## DNS 구성 요소

| 구성 요소 | 다른 이름 | 역할 |
| --- | --- | --- |
| **도메인 네임 스페이스** (Domain Name Space) | — | 도메인 이름을 분산 저장하는 규칙 |
| **네임 서버** (Name Server) | 권한 있는 DNS 서버 | 해당 도메인 이름의 IP 주소를 찾음 |
| **리졸버** (Resolver) | 권한 없는 DNS 서버 | 클라이언트 요청을 네임 서버에 전달하고, 찾은 정보를 응답 + 캐싱 |

---

## DNS 서버 종류

도메인 수가 매우 많기 때문에, DNS 서버는 하나가 아니라 **계층화하여 단계적으로** 처리한다.

- **Root DNS Server**
  ICANN(국제 인터넷 관리기구)이 직접 관리하는 최상위 서버. TLD DNS 서버의 IP들을 저장해 두고 안내한다.

- **TLD(최상위 도메인) DNS Server**
  도메인 등록 기관(Registry)이 관리하는 서버. Authoritative DNS 서버 주소를 저장해 두고 안내한다. 어떤 도메인 묶음이 어떤 Authoritative DNS Server에 속하는지 알 수 있는 이유는, 도메인 판매 업체(Registrar)의 DNS 설정이 변경되면 그 정보가 도메인 등록 기관으로 전달되기 때문이다.

- **Authoritative DNS Server**
  실제 개인 도메인과 IP 주소의 관계가 기록·저장·변경되는 서버. 일반적으로 도메인/호스팅 업체의 '네임서버'를 말하지만, 개인이 직접 DNS 서버를 구축한 경우에도 여기에 해당한다.

- **Recursive DNS Server**
  인터넷 사용자가 가장 먼저 접근하는 DNS 서버. 위 3개 서버를 매번 거치지 않도록, 한 번 조회한 데이터를 일정 기간(**TTL**) 동안 캐시로 저장해 둔다. KT/LG/SKT 등 ISP의 DNS 서버가 있고, 브라우저 우회 용도로 많이 쓰는 구글 DNS, 클라우드플레어 같은 Public DNS 서버도 여기 속한다.

> **정리**: 브라우저는 캐시가 저장된 **Recursive 서버**를 사용하고, 실제 네임서버 역할을 하는 곳은 **Authoritative 서버**다.

## DNS 동작 원리

`NAVER.com` 검색을 예로, KT를 통신사로 가정한 전체 흐름이다.

1. 브라우저에서 `NAVER.com`을 검색하고, 사용 중인 통신사 **KT DNS 서버**에게 도메인에 해당하는 IP 주소를 요청한다.
2. ISP 서버는 캐시 데이터가 없음을 확인하고, **루트 DNS 서버**에게 어디로 가야 하는지 요청한다. *(캐시가 있다면 8번으로 건너뜀)*
3. 루트 서버는 TLD DNS 서버 주소만 관리하므로, `.com` 도메인을 보고 **COM 최상위 도메인을 관리하는 TLD DNS 서버 주소**를 안내한다.
4. ISP 서버는 **COM 서버**에게 어디로 가야 하는지 요청한다.
5. COM 서버는 해당 도메인이 **가비아 DNS 서버**에서 관리되고 있음을 확인하고 안내한다.
6. ISP 서버는 **가비아 서버**에게 또 요청한다.
7. 가비아 서버는 `NAVER.com = 12.123.123.123` 정보를 확인하고 이 IP를 알려준다. 동시에 ISP 서버는 이 정보를 **캐시로 기록**해 둔다.
8. ISP 서버는 브라우저에게 알아낸 `12.123.123.123` 주소를 안내한다.
9. 브라우저는 이 IP 주소를 가진 **호스팅 서버**에게 웹사이트를 출력하라고 요청한다.
10. 웹사이트가 보인다!

---

## 핵심 정리

- 도메인 → IP 변환은 **Root → TLD → Authoritative** 순서의 계층 구조로 이뤄진다.
- 매번 전 과정을 거치면 느리기 때문에, **Recursive 서버의 캐시(TTL)**로 속도를 높인다.
- 도메인의 실제 IP 정보가 '진짜로' 저장되는 곳은 **Authoritative DNS 서버**다.
