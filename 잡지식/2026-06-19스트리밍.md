# TIL: 라이브 스트리밍 기술의 동작 원리

> 트위치/치지직 같은 실시간 라이브 방송이 어떤 단계를 거쳐 시청자에게 전달되는지, 그리고 직접 미니 스트리밍 서버를 띄워본 기록.

## 전체 흐름

라이브 스트리밍은 크게 4단계로 이루어진다.

```
방송인 컴퓨터  →  미디어 서버  →  CDN  →  시청자
 (인코딩)       (변환·포장)    (배포)    (재생)
```

| 단계 | 하는 일 | 비유 |
| --- | --- | --- |
| 인코딩 | 원본 영상을 코덱으로 압축 | 짐을 박스에 포장 |
| 업로드 | 압축 영상을 서버로 전송 (RTMP) | 택배 접수 |
| 미디어 서버 | 시청용 포맷으로 재변환 (HLS) | 물류센터 |
| CDN | 전 세계 서버에 복사 후 배포 | 전국 택배 지점 |

## 각 단계 정리

### 1. 인코딩 (Encoding)

- 원본 화면은 초당 수 기가비트 수준이라 그대로 전송이 불가능하다.
- `H.264` 같은 코덱으로 압축하면 1080p 기준 약 5Mbp까지 줄어든다.
- `OBS` 같은 프로그램이 이 역할을 담당한다.

### 2. 업로드 (RTMP)

- `RTMP`는 압축된 영상을 미디어 서버로 끊김 없이 전송하는 프로토콜이다.
- 트위치의 스트림 키(stream key) 가 이 전송 통로의 인증 역할을 한다.
- 방송인 1명 → 서버 1대 구간이라 안정성이 핵심이다.

### 3. 미디어 서버 (HLS 변환)

- `HLS`는 영상을 2~6초짜리 작은 조각(`.ts`)으로 자르고, 조각 목록표(`.m3u8`)와 함께 제공하는 방식이다.
- 플레이어는 `.m3u8`을 계속 새로고침하며 새 조각을 받아 이어 붙여 재생한다.
- 조각을 모아 보내는 구조 때문에 5~30초의 지연이 발생한다.
- 트위치/치지직이 2~5초로 빠른 것은 `LL-HLS` 같은 저지연 기술을 따로 적용하기 때문이다.

### 4. CDN

- 시청자가 많아지면 서버 1대로는 감당이 안 된다.
- 영상 조각을 전 세계 서버에 복사해두고, 시청자는 가장 가까운 서버에서 받아간다.
- 대형 스트리밍 서비스의 핵심 인프라다.

## 직접 해보기: 로컬 미니 트위치

준비물: **OBS Studio**, **Docker Desktop** (둘 다 무료)

### 1) 미디어 서버 띄우기

```bash
docker run -d -p 1935:1935 -p 8080:8080 --name mini-twitch alqutami/rtmp-hls
```

- `1935` 포트: OBS가 영상을 밀어 넣는 RTMP 입구
- `8080` 포트: 시청자가 HLS 조각을 받아가는 출구
- `docker ps`로 컨테이너가 떠 있는지 확인

### 2) OBS 송출 설정

설정 → 방송 탭에서:

- **서비스**: 사용자 지정(Custom)
- **서버**: `rtmp://localhost:1935/live`
- **스트림 키**: `test` (아무거나 가능)

소스에 화면 캡처를 추가하고 **방송 시작**을 누른다.

### 3) 웹 플레이어로 시청하기

`player.html`로 저장 후 브라우저로 열기:

```html
<!DOCTYPE html>
<html>
<body>
  <video id="video" controls autoplay muted style="width: 100%"></video>
  <script src="https://cdn.jsdelivr.net/npm/hls.js@latest"></script>
  <script>
    const video = document.getElementById('video');
    const src = 'http://localhost:8080/hls/test.m3u8';

    if (Hls.isSupported()) {
      const hls = new Hls();
      hls.loadSource(src);
      hls.attachMedia(video);
    } else {
      video.src = src; // 사파리는 HLS 네이티브 지원
    }
  </script>
</body>
</html>
```

- URL 끝의 `test.m3u8`이 조각 목록표이며, 스트림 키에 따라 파일명이 바뀐다. (사실상 방송 채널 개념)

### 4) 라이브 지연

- OBS 앞에서 손을 흔들면 플레이어에는 **10~30초 늦게** 나온다 → HLS의 구조적 지연.
- OBS 설정 → 출력 → 키프레임 간격을 `1`초로 바꾸면 지연이 줄어든다.

### 트러블슈팅

- 영상이 안 뜨면 방송 시작 후 **10초 대기** 후 새로고침 (첫 조각 생성 시간 필요)
- `8080` 포트 충돌 시 `-p 8081:8080`으로 변경하고 URL도 함께 수정
- OBS "연결 실패" 시 Docker 컨테이너 상태부터 확인


## 배운 점

- 인코더·프로토콜을 바닥부터 구현할 필요 없이, 오픈소스를 조립하면 된다.
- 영상 파이프라인과 웹 서비스(회원가입, 방송 목록, 스트림 키 발급, 채팅)는 분리되어 있다. 실제 트위치도 동일한 구조.
- 진짜 어려워지는 단계: 화질 자동 조절(ABR), 저지연, 동시 시청자 수천 명 처리.
